# 日志组件使用说明

## 概述

Daily Discover 项目采用分层日志架构，包含三个独立的日志系统：
- **HTTP API 日志系统**：专门记录HTTP层面的API调用信息
- **服务层日志系统**：专门记录Service层方法的调用和异常信息
- **业务日志追踪系统**：专门记录业务层面的方法调用和操作

## 组件架构

### 1. HTTP API 日志系统

```
ApiLogAspect (切面) → ApiLogger (日志记录器) → ApiLogConfig (配置)
```

**核心组件：**
- `ApiLogAspect` - 自动拦截Controller方法的AOP切面
- `ApiLogger` - HTTP API日志记录器
- `ApiLog` - 自定义注解（**仅提供接口描述功能**）
- `ApiLogConfig` - 配置类（**仅支持排除配置**）

### 2. 服务层日志系统

```
ServiceLogAspect (切面) → 自动记录服务层方法调用和异常
```

**核心组件：**
- `ServiceLogAspect` - 自动拦截Service层方法的AOP切面
- `GlobalExceptionHandler` - 全局异常处理器（增强服务层异常处理）

### 3. 业务日志追踪系统

```
LogTracer (业务日志追踪工具)
```

**核心组件：**
- `LogTracer` - 业务层面的日志追踪工具类

## HTTP API 日志系统

### 功能特性

- ✅ **自动拦截**：自动拦截所有Controller包下的方法
- ✅ **HTTP层面记录**：记录URL、HTTP方法、客户端IP、状态码等
- ✅ **性能监控**：记录方法执行时间
- ✅ **异常追踪**：记录异常信息
- ✅ **配置化控制**：支持按包、类、方法级别排除

### 使用方式

#### 1. 自动拦截（推荐）

无需任何配置，所有Controller包下的方法会自动被拦截：

```java
@RestController
@RequestMapping("/api/users")
public class UserController {
    
    @GetMapping("/{id}")
    public User getUser(@PathVariable String id) {
        // 该方法会自动被ApiLogAspect拦截并记录日志
        return userService.getUserById(id);
    }
}
```
 
#### 2. 自定义注解（可选，功能有限）

`@ApiLog`注解目前**仅支持提供自定义接口描述**，其他配置属性（如`logRequest`、`logResponse`等）**尚未实现**：

```java
@PostMapping("/login")
@ApiLog(value = "用户登录接口")  // 仅value属性生效
public ResponseEntity<Map<String, Object>> login(@RequestBody LoginRequest request) {
    // 日志会显示"用户登录接口"而不是默认的方法名
    return authService.login(request);
}
```

**注意**：注解中的`logRequest`、`logResponse`、`logExecutionTime`、`logException`等配置属性目前**没有实际效果**。

### 配置说明

`ApiLogConfig`配置类**功能有限**，仅支持基础的排除配置：

```yaml
api:
  log:
    enabled: true                    # 是否启用API日志
    exclude-packages: ""            # 排除的包路径（逗号分隔）
    exclude-classes: ""             # 排除的类名（逗号分隔）
    exclude-methods: ""             # 排除的方法名（逗号分隔）
```

**配置限制**：
- 不支持细粒度的日志控制（如是否记录请求参数、响应结果等）
- 不支持与`@ApiLog`注解的配置属性联动
- 仅提供基础的包/类/方法排除功能

## 业务日志追踪系统

### 功能特性

- ✅ **业务方法追踪**：记录业务方法的入参和出参
- ✅ **数据库操作追踪**：记录SQL查询和结果
- ✅ **业务API调用追踪**：记录业务层面的API调用
- ✅ **性能追踪**：记录方法执行时间并分级
- ✅ **异常追踪**：记录业务异常信息
- ✅ **链式追踪**：支持链式调用的上下文追踪

### 使用方式

#### 1. 业务方法追踪

```java
@Service
public class UserService {
    
    public User createUser(User user) {
        // 追踪业务方法调用
        LogTracer.traceBusinessMethod(user, createdUser);
        
        User createdUser = userRepository.save(user);
        return createdUser;
    }
    
    public void updateUser(User user) {
        // 仅追踪入参
        LogTracer.traceBusinessMethodWithParams(user);
        
        userRepository.update(user);
    }
}
```

#### 2. 数据库操作追踪

```java
@Repository
public class UserRepository {
    
    public User findById(String id) {
        String sql = "SELECT * FROM users WHERE id = ?";
        User user = jdbcTemplate.queryForObject(sql, new Object[]{id}, userMapper);
        
        // 追踪数据库查询
        LogTracer.traceDatabaseQuery(sql, id, user);
        
        return user;
    }
}
```

#### 3. 性能追踪

```java
@Service
public class OrderService {
    
    public Order createOrder(OrderRequest request) {
        long startTime = System.currentTimeMillis();
        
        // 复杂的业务逻辑...
        Order order = processOrder(request);
        
        // 性能追踪
        LogTracer.traceBusinessPerformance(startTime);
        
        return order;
    }
}
```

#### 4. 链式追踪

```java
@Service
public class OrderProcessingService {
    
    public void processOrder(String orderId) {
        // 创建追踪上下文
        LogTracer.TraceContext context = LogTracer.createTraceContext("order_" + orderId)
            .add("orderId", orderId)
            .add("step", "开始处理");
        
        context.trace("订单处理开始");
        
        // 处理步骤1
        context.add("step", "库存检查").trace("库存检查");
        
        // 处理步骤2
        context.add("step", "支付验证").trace("支付验证");
        
        // 处理步骤3
        context.add("step", "订单创建").trace("订单创建完成");
    }
}
```

## 组件价值分析

### ApiLog注解的实际价值
- ✅ **提供接口描述**：可以为接口方法提供更友好的描述文本
- ❌ **配置功能未实现**：`logRequest`、`logResponse`等配置属性在切面中未被使用
- ⚠️ **功能有限**：目前仅相当于一个描述标签，没有实际的配置控制能力

### ApiLogConfig的实际价值
- ✅ **基础排除功能**：支持按包、类、方法排除日志记录
- ❌ **细粒度控制缺失**：无法控制是否记录请求参数、响应结果等细节
- ⚠️ **配置简单**：仅提供基础的开关和排除功能

## 最佳实践

### 1. 分层使用
- **HTTP层面**：使用ApiLogAspect记录HTTP API调用（功能相对基础）
- **业务层面**：使用LogTracer记录业务逻辑和数据库操作（功能更丰富）

### 2. 配置建议
- **大多数情况下**：使用默认的自动拦截即可，无需使用`@ApiLog`注解
- **需要友好描述时**：可以使用`@ApiLog(value = "描述")`提供接口描述
- **排除特定接口时**：使用`exclude-packages`、`exclude-classes`、`exclude-methods`配置

### 3. 性能考虑
- 生产环境可以使用排除配置减少高频接口的日志记录

### 4. 环境配置

**开发环境**：
```yaml
api:
  log:
    enabled: true
logging:
  level:
    com.dailydiscover.common.logging: DEBUG
```

**生产环境**：
```yaml
api:
  log:
    enabled: true
    exclude-packages: "com.dailydiscover.controller.health"
logging:
  level:
    com.dailydiscover.common.logging: INFO
```

## 故障排除

### 1. 日志不生效
- 检查`api.log.enabled`配置
- 确认组件扫描包包含`com.dailydiscover.common`
- 检查AOP是否启用（`@EnableAspectJAutoProxy`）

### 2. 性能问题
- 检查是否记录了过多不必要的日志
- 考虑使用排除配置过滤高频接口

### 3. 日志格式问题
- 确认日志级别设置正确
- 检查日志配置文件格式

## 服务层日志系统

### 功能特性

- ✅ **自动拦截**：自动拦截所有Service包下的方法
- ✅ **方法调用追踪**：记录服务方法的调用和执行时间
- ✅ **异常自动处理**：与全局异常处理器配合，提供详细的异常上下文
- ✅ **零侵入性**：无需修改现有代码，自动生效
- ✅ **性能优化**：使用DEBUG级别日志，生产环境可关闭

### 使用方式

#### 1. 自动拦截（推荐）

无需任何配置，所有Service包下的方法会自动被拦截：

```java
@Service
public class ProductServiceImpl implements ProductService {
    
    @Override
    public ProductBasicInfoDTO findBasicInfoById(Long id) {
        // 该方法会自动被ServiceLogAspect拦截并记录日志
        // 无需手动添加try-catch，异常会自动被全局异常处理器处理
        return productMapper.findBasicInfoById(id);
    }
}
```

#### 2. 与全局异常处理器配合

服务层异常会被`GlobalExceptionHandler`捕获并记录详细的上下文信息：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    @ExceptionHandler(Exception.class)
    public ResponseEntity<Void> handleException(Exception e) {
        // 自动识别服务层方法并记录异常上下文
        log.error("服务层异常 - 方法: {}, 异常: {}", serviceMethod, e.getMessage(), e);
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).build();
    }
}
```

🔧 服务方法调用 | ProductServiceImpl.findBasicInfoById | 2025-02-24 10:30:15 | 服务方法 ProductServiceImpl.findBasicInfoById 执行成功 - 耗时: 23ms

### 性能优化

- **开发环境**：DEBUG级别日志开启，便于调试
- **生产环境**：DEBUG级别日志关闭，零性能开销
- **异常处理**：只在异常发生时记录，正常流程无开销

## 组件价值分析

### 服务层日志系统的价值
- ✅ **统一异常处理**：避免服务层重复的try-catch代码
- ✅ **详细异常上下文**：自动识别异常发生的具体服务方法
- ✅ **性能优化**：正常流程几乎零开销，异常时提供完整信息
- ✅ **代码简洁**：服务层代码更清晰，专注于业务逻辑

### 与现有组件的配合
- **ApiLogAspect**：处理HTTP层面日志
- **ServiceLogAspect**：处理服务层面日志
- **LogTracer**：处理业务层面详细追踪
- **GlobalExceptionHandler**：统一异常处理

## 最佳实践

### 1. 分层使用（推荐）
- **HTTP层面**：使用ApiLogAspect记录HTTP API调用
- **服务层面**：使用ServiceLogAspect记录服务方法调用
- **业务层面**：使用LogTracer记录详细业务逻辑
- **异常处理**：统一使用GlobalExceptionHandler

### 2. 代码重构建议
- **移除服务层try-catch**：让AOP和异常处理器自动处理
- **保持代码简洁**：服务层专注于业务逻辑实现
- **统一日志格式**：使用分层的日志架构

### 3. 环境配置

**开发环境**：
```yaml
logging:
  level:
    com.dailydiscover.common.logging: DEBUG
    com.dailydiscover.service: DEBUG
```

**生产环境**：
```yaml
logging:
  level:
    com.dailydiscover.common.logging: INFO
    com.dailydiscover.service: INFO
```

### 4. 性能优化建议

1. **使用DEBUG级别**：生产环境可关闭服务层详细日志
2. **避免复杂切面逻辑**：切面中只做简单的日志记录
3. **异常处理优化**：只在异常发生时记录详细信息

通过这种三层日志架构（HTTP层 + 服务层 + 业务层），可以更清晰地分离不同层面的日志记录，便于问题排查和系统监控，同时保持代码的简洁性和性能优化。
